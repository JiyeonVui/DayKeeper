# DayKeeper — Project Architecture

> Generated by reading the source. Statements marked `(unverified)` are inferences not fully confirmed in code.

## 1. Overview

- **Name:** DayKeeper (`com.jiyeon.daykeeper`)
- **Purpose:** A weekly activity planner. Users create recurring activities (e.g. "Gym", "Korean study") that repeat on selected weekdays, each with a start/end time, a category, an optional note, and a reminder toggle. The current day's activities are shown on a timeline.
- **Problem it solves:** Organizing a repeating weekly routine and catching time clashes — saving an activity that overlaps an existing one on a shared weekday is blocked with a conflict warning.
- **UI language:** Vietnamese (all user-facing strings and most code comments are Vietnamese).
- **Development stage:** Early MVP / in active development. Timeline + Add/Edit + conflict detection are functional. Summary and Settings are placeholder screens. No reminders/notifications, no tests beyond scaffolding, manual dependency wiring (Hilt planned per comment).

## 3. Architecture

Layered MVVM with a thin repository. Roughly Clean-ish but pragmatic (no domain/usecase layer).

| Layer | Location | Responsibility |
|-------|----------|----------------|
| **Data / local** | `data/local/` | Room `@Entity` (`ScheduleItem`), `ScheduleDao`, `AppDatabase`, `Converters`, weekday bitmask helpers |
| **Data / repository** | `data/` | `ScheduleRepository` wraps the DAO; `ConflictChecker` (pure domain logic) |
| **Presentation (ViewModels)** | `ui/*/...ViewModel.kt` | `TimelineViewModel`, `AddEditViewModel` expose `StateFlow`; manual `ViewModelProvider.Factory` per VM |
| **UI (Compose)** | `ui/**` | Stateless screens + components; state hoisted to hosts/ViewModels |
| **Theme** | `ui/theme/` | Design tokens (`DayKeeperColors`, `DayKeeperType`), `DayKeeperTheme` |
| **Util** | `util/` | Pure formatting helpers (time, date) |

**Communication flow:**
- `Composable` → `ViewModel` method calls; `ViewModel` → `Repository` → `Dao` → Room.
- Reads are reactive: DAO returns `Flow`, ViewModel converts to `StateFlow` via `stateIn`/`flatMapLatest`, UI collects with `collectAsStateWithLifecycle`.
- Writes are `suspend` calls launched in `viewModelScope`.
- The Add/Edit form state is hoisted in a **stateful host** (`AddEditHost`) using `remember(existingItem)`, separate from the **stateless** `AddEditScreen` that only renders.

**Pattern in use:** MVVM + Repository pattern + unidirectional data flow (stateless composables, hoisted state).

## 4. Data Model

### `ScheduleItem` (Room entity, table `schedule_items`)

| Field | Type | Meaning |
|-------|------|---------|
| `id` | `Long` (autoGenerate) | primary key, `0` = new/unsaved |
| `title` | `String` | activity name |
| `note` | `String = ""` | optional note |
| `startMinute` | `Int` | **start time as minutes-from-midnight** (e.g. 420 = 07:00) |
| `endMinute` | `Int` | **end time as minutes-from-midnight** (e.g. 480 = 08:00) |
| `daysOfWeek` | `Int` | **bitmask of repeat days** (see Weekday bits below) |
| `reminderEnabled` | `Boolean = true` | reminder on/off (UI only — no notification engine yet) |
| `reminderOffsetMin` | `Int = 0` | reminder lead time; defined but **not surfaced in UI** |
| `category` | `ActivityCategory = STUDY` | drives the display color |

### `ActivityCategory` (enum)

`REST` ("Nghỉ ngơi"), `STUDY` ("Học tập"), `SPORT` ("Thể thao"), `WORK` ("Công việc"). `DEFAULT = STUDY`. Each carries a Vietnamese `label`. **Color is always derived from the category — never chosen manually** (extension fns in `CategoryColors.kt`).

### Special conventions

- **Time = minutes-from-midnight `Int`.** Formatting via `Int.toHourMinute()` (`420 → "07:00"`). Picker minutes are snapped to multiples of 5 (`snapTo5` per spec).
- **Weekday = bitmask `Int`.** `Weekday` object: `MON=1, TUE=2, WED=4, THU=8, FRI=16, SAT=32, SUN=64`. Helpers: `DayOfWeek.toBit()`, `todayBit()`, `ScheduleItem.repeatsOn(dayBit)`, `daysToText(mask)` → "T2, T4".
- `TimeConflict(conflictingItem, overlappingDays: Int)` — value type returned by `ConflictChecker`.

## 6. Core Flows

### A. View timeline for a day
1. `MainActivity` builds `AppDatabase → ScheduleRepository → TimelineViewModelFactory`, sets `MainScreen`.
2. `TimelineViewModel` starts with `_selectedDay = todayBit()`.
3. `items` = `_selectedDay.flatMapLatest { repo.itemsForDay(it) }` → DAO query `WHERE (daysOfWeek & :dayBit) != 0 ORDER BY startMinute`.
4. `TimelineScreen` collects `selectedDay` + `items`; `DayTabRow` shows Mon–Sun of the current week, tapping a tab calls `selectDay(bit)`, re-triggering the flow.
5. Each `ScheduleItem` renders as `ScheduleItemRow` (time + duration, category color bar, title/note, reminder bell icon).

### B. Add/Edit + conflict-gated save (the key flow)
1. FAB → `MainScreen` sets `showAddEdit=true`, `editingItemId=null` (create) **or** item id (edit). Editing looks up the item from the **currently selected day's** `items` list.
2. `AddEditHost` hoists form fields with `remember(existingItem)` (resets when the item changes). Create defaults: 07:00–08:00, `category=STUDY`, reminder on.
3. User edits name/category/times (via `TimePickerDialog`, minutes snapped to 5)/repeat-day chips/reminder/note. All updates are **immutable** (Set add/remove produces a new set; `copy()` for the item).
4. On "Lưu": host does light validation (`name` non-blank AND `endMinute > startMinute`), builds a `ScheduleItem` via `copy()` (preserving `id` & `reminderOffsetMin` when editing), calls `AddEditViewModel.trySave`.
5. `trySave` loads all items (`getAllOnce`), runs `ConflictChecker.findConflicts`:
   - Skips the candidate's own id; requires a **shared weekday bit** (`daysOfWeek and daysOfWeek != 0`); overlap test is **half-open**: `start < otherEnd && otherStart < end`.
   - **If conflicts found → does NOT save**, publishes them to `conflicts` StateFlow → `ConflictDialog` lists the clashing activities and which days overlap.
   - **If none → `repo.save` (upsert) then `onSuccess()` → close.**

## 7. Conventions & Constraints

- **Package root:** `com.jiyeon.daykeeper`. Organized by feature/domain (`timeline`, `addedit`, `settings`...), not by type.
- **Naming:** `*Screen` = stateless UI; `*Host` = stateful wrapper; `*ViewModel` + matching `*ViewModelFactory`; `components/` subfolders for sub-composables.
- **Color rule (deliberate):** Never hardcode hex/sp inline. Theme-aware tokens come from `MaterialTheme.colorScheme.*`; tokens without a Material slot (TextTertiary, category colors, soft fills) come from `DayKeeperColors`. Category color is always *derived* from `ActivityCategory`, never picked.
- **Typography:** Only two font weights — 400 Normal and 500 Medium (`TextStyle.medium` extension). No color set inside `TextStyle`; color passed at call site.
- **Immutability:** Enforced everywhere — toggling a day set or editing a field always yields a new copy (matches the global coding-style rule).
- **Stateless composables:** Local state limited to genuinely local concerns (dialog `showInput`, host `remember`s).
- **File size:** Spec targets composables small, files < 800 lines (largest is `AddEditScreen.kt` at 433).
- **Things to avoid:** dynamic/system colors; inline literal colors/text sizes; mutation; (currently) adding Navigation-Compose unless intentionally replacing the pager.

## 9. Glossary / Domain Knowledge

- **minutes-from-midnight:** All times are stored as an `Int` count of minutes since 00:00 (420 = 07:00). Not a `LocalTime`.
- **day bitmask (`daysOfWeek`):** One `Int` encodes which weekdays an activity repeats on, OR-ing `Weekday` constants (Mon=1 … Sun=64). A query `(daysOfWeek & dayBit) != 0` finds items active on a given day.
- **category-derived color:** Display color is a pure function of `ActivityCategory` (bar color, plus soft-fill + on-soft-fill for the selected chip in Add/Edit).
- **half-open overlap:** Two activities clash if `start < otherEnd && otherStart < end` — touching edges (one ends exactly when another starts) do **not** conflict.
- **soft fill:** The light category-tinted background used for the *selected* category chip on the Add/Edit form (`CategoryXSoft` / `OnCategoryXSoft` tokens).
- **Vietnamese day labels:** UI shows T2..T7, CN (Mon..Sun); full dates like "Thứ Hai, 22 tháng 6".
